01 / OVERVIEW

Practical Embedded Linux work with a clear MCU integration path.

BossMonitor-AI was created to turn camera input, motion detection, event capture and future hardware control into one explainable embedded system.

WHY THIS PROJECT EXISTS

Build practical Embedded Linux experience

The project uses Armbian Linux on Orange Pi Zero to practice camera access, Linux device workflow and application-level automation.

SYSTEM BOUNDARY

Connect Linux applications with MCU firmware

The current software is structured so a UART link to STM32 can be added later without rewriting the camera pipeline.

PRACTICE AREA

Capture events from camera input

The working scope focuses on USB camera image capture, motion detection, saving event images and controlling detection delay.

03 / CURRENT IMPLEMENTATION

Working functions kept intentionally small and testable.

These are the current software capabilities described as implemented prototype behavior, not final product claims.

CAMERA CAPTURE

USB camera image capture

Captures frames from a Logitech C270 camera connected to the Orange Pi Zero.

VISION PIPELINE

Motion detection

Uses Python and OpenCV to identify motion from camera input.

EVENT OUTPUT

Event image saving

Saves images when motion is detected so each event leaves reviewable evidence.

CAPTURE CONTROL

Detection delay control

Uses a configurable delay to avoid repeated image captures from one continuous motion event.

05 / ENGINEERING CHALLENGES

Grounded problems from building the prototype.

The useful work here is in making Linux camera input reliable and leaving room for MCU integration.

Camera setup on Linux

Setting up the USB camera on Linux and confirming that the device is visible to the system.

Camera access verification

Verifying camera access with Linux tools before relying on the Python application path.

Repeated capture control

Avoiding repeated image captures from one continuous motion event with configurable delay behavior.

Future UART boundary

Structuring the software so STM32 UART integration can be added later as a separate control layer.

06 / WHAT I LEARNED

Evidence of learning across the Linux-to-MCU path.

The project connects low-level workflow practice with a future hardware-control direction.

Linux camera device workflow

How to reason about USB camera access from the Linux system layer before application development.

OpenCV motion-detection pipeline

How camera frames can be processed into motion events with a basic OpenCV workflow.

Event-driven image capture

How to trigger image saving from detected events instead of continuously saving frames.

Linux-to-MCU integration boundary

How to plan a UART boundary where Linux application events can later drive STM32 firmware behavior.
